## Deviations From Original Plan

Captured as they happened so future me (or another agent) isn't surprised.

- **CLI surface: `podsave save <url>` instead of `podsave <url>`.** Plan wrote bare-URL-at-root; shipped it as a subcommand for unambiguous parsing against `queue`, `init`, `retry`. `--dry-run` stayed as a flag on `save`. (P2)
- **`PODSAVE_HOME` env var for state-dir override.** Not in the plan, added for test isolation (tmp home per test). Also useful as a user escape hatch. `paths.get_home()` reads it. (P1)
- **`init` also creates `transcripts/` and `tmp/` dirs.** Plan only listed `config.toml` + `queue.txt` + `processed.jsonl`. Added the other dirs so first real `save` run doesn't hit "directory not found". (P1)
- **Added `podsave version` command.** Single-command Typer apps collapse the lone command into the root, which broke help-output tests. Added `version` to keep multi-command shape; useful anyway. (P0)
- **`@handle_errors` decorator on commands.** `main()`'s try/except is bypassed by `CliRunner.invoke(app, ...)` in tests, so PodsaveError leaked as unhandled. Added a tiny decorator that catches PodsaveError and calls `_fail()`. Every user-facing command wears it. (P2)
- **Env-var overrides for config values.** Plan spec had keys in `config.toml`. Added `PODSAVE_OPENAI_API_KEY` / `PODSAVE_ASSEMBLYAI_API_KEY` / `PODSAVE_VAULT_PATH` / `PODSAVE_EXTRACTION_MODEL` that override the file. Makes CI/integration tests trivial and matches how most CLIs behave. (P1)
- **`yt-dlp --dump-single-json` (not plain `-J`).** Same output but the long flag is explicit and less likely to break if yt-dlp ever re-uses `-J`. (P2)
- **No ffmpeg dependency.** Plan implicitly assumed mp3 extraction. Skipped it: `yt-dlp -f bestaudio` downloads the native stream (typically webm/m4a), AssemblyAI accepts both. One fewer install step, one fewer moving part. (P3)
- **AssemblyAI `speech_models=["universal-3-pro"]` pinned.** API now rejects calls without an explicit model and the SDK's default is stale. Also the catalyst for switching the STT rate constant from `$0.37/hr` (universal-2) to `$0.21/hr` (universal-3-pro). (P3)
- **Raw JSON transcript persisted as-is.** Plan considered a normalized transcript shape up front. Deferred — writing the raw AssemblyAI response keeps all fields (utterances, words, confidence, etc.) available for Phase 4 without a migration if the extraction prompt wants more context. Meta sidecar is still the typed VideoMeta. (P3)

None of these change the v1 scope or the user-visible promise. All worth preserving.

## Goal

`podsave save <url>` consumes a YouTube video and lands a single, well-structured markdown note in `~/obsd/Resources/Videos/` containing the ten best insights, quotes, and spicy takes from the episode. All timestamps in quotes link back to the exact YouTube moment. Transcripts are cached so extraction is cheap to retry and rework.

## Non-Goals (v1)

- Podcast RSS, Apple/Spotify, arbitrary mp3 URLs — YouTube only.
- Playlists — explicit error, not silent expansion.
- Local STT / local LLM — hooks for future, no implementation.
- Webhook-based STT — poll for v1, it's fine.
- Parallel queue drain — sequential to avoid rate-limit juggling.
- Re-processing detection across runs — re-running just bumps version; no diffing.
- Web UI, TUI, scheduled background runs.

## Architecture Sketch

```
┌─ cli.py (Typer) ──────────────────────────────────────────┐
│  podsave <url>                                             │
│  podsave queue add|list | --drain                          │
│  podsave retry <video_id>                                  │
│  podsave init                                              │
│  podsave --dry-run <url>                                   │
└──────────────┬─────────────────────────────────────────────┘
               │
               ▼
┌─ pipeline/ ────────────────────────────────────────────────┐
│  download.probe(url)  → VideoMeta (no download, yt-dlp -J) │
│  download.fetch(url)  → mp3 path                           │
│  transcribe.run(mp3)  → AssemblyAI JSON (diarized)         │
│  extract.run(transcript, meta) → ExtractionResult (top-10) │
│  render.note(meta, extraction) → markdown string           │
└──────────────┬─────────────────────────────────────────────┘
               │
               ▼
┌─ storage/ ─────────────────────────────────────────────────┐
│  ~/.podsave/config.toml        API keys, vault path        │
│  ~/.podsave/queue.txt          one URL per line            │
│  ~/.podsave/transcripts/       <video_id>.json + .meta.json│
│  ~/.podsave/processed.jsonl    append-only run log         │
│  ~/obsd/Resources/Videos/      notes (vault)               │
└────────────────────────────────────────────────────────────┘
```

The pipeline stages are pure-ish functions over data models. The CLI wires them together. Storage is plain files — no SQLite, no ceremony.

## Data Model Sketch

Defined in `src/models.py` as Pydantic models:

- **`VideoMeta`** — `video_id`, `url`, `title`, `channel`, `published` (date), `duration_sec`
- **`Utterance`** — `speaker`, `start_ms`, `end_ms`, `text`, `confidence`
- **`Transcript`** — `video_id`, `language`, `full_text`, `utterances: list[Utterance]`, `raw: dict` (the full AssemblyAI response, stored for future reprocessing)
- **`Insight`** — `rank: int` (1-10), `kind: Literal["insight", "quote", "spicy_take"]`, `text`, `speaker: str | None`, `timestamp_ms: int | None` (only for quotes)
- **`ExtractionResult`** — `insights: list[Insight]`, `model`, `prompt_version: str`, `extracted_at`
- **`RunRecord`** — the log line: `url`, `video_id`, `processed_at`, `version`, `note_path`, `transcript_path`, `cost_usd: dict[str, float]` (broken down by service), `duration_sec`, `status: Literal["complete", "transcript_only", "failed"]`

## Pricing Math (as of Jan 2026 knowledge cutoff — verify before locking)

Numbers below are estimates for `--dry-run`. Must be validated with one real test run against short, medium, and long videos before we trust the cap/preview logic.

**Landed in code** at `src/utils/cost.py` as module constants:
- `ASSEMBLYAI_PER_HOUR_USD = 0.37`
- `OPENAI_INPUT_PER_MILLION_USD = 2.50`, `OPENAI_OUTPUT_PER_MILLION_USD = 10.0`
- `TOKENS_PER_MINUTE_AUDIO = 200`, `OUTPUT_TOKENS_PER_RUN = 500`

Recalibrate these after Phase 3's first real AssemblyAI + OpenAI bills.

**AssemblyAI (with diarization, Universal tier)**
- ~$0.37/hour of audio at time of cutoff
- Duration in seconds → `(duration_sec / 3600) * 0.37`

**OpenAI extraction**
- Input tokens scale with transcript length. Rule of thumb: ~150 words/min spoken × ~1.3 tokens/word ≈ **~200 tokens per minute of audio**.
- So 1h video → ~12k input tokens; 3h Rogan → ~36k tokens.
- Output is bounded (10 insights, ~500 tokens).
- At GPT-4-class pricing (~$2.50/1M input, ~$10/1M output): 1h ≈ $0.03–$0.04; 3h ≈ $0.09–$0.10.

**Total preview for `--dry-run`**: AssemblyAI dominates. 1h ≈ $0.38, 3h ≈ $1.15. These are small enough that the cap is a safety belt, not a budget.

**Hard cap**: refuse > 4h without `--force`. (Adjust once we see real bills.)

## How We Get Duration Without Downloading

`yt-dlp --dump-json <url>` (or `yt-dlp -J`) returns full metadata — including `duration`, `title`, `uploader` (channel), `upload_date`, `id` — without downloading the audio stream. This is what `download.probe()` uses. Cheap, fast, safe for `--dry-run` and for populating `VideoMeta` before we commit to the download.

## Retry Without Re-Hitting YouTube

Next to every cached transcript at `~/.podsave/transcripts/<video_id>.json`, write a sibling `~/.podsave/transcripts/<video_id>.meta.json` containing the full `VideoMeta`. When `podsave retry <video_id>` runs, it loads both JSONs from disk and skips the download + STT stages entirely — extraction only. This also means retries don't need internet access for yt-dlp. (The `processed.jsonl` log has metadata too, but a sidecar is simpler — one lookup, one file.)

## Failure Modes & Recovery

The pipeline has three expensive stages. Each stage's output is persisted before moving to the next, so a failure late in the pipeline never wastes earlier work.

| Stage | Persists | If fails after | Recovery |
|-------|----------|----------------|----------|
| `probe` | nothing | — | cheap, just re-run |
| `download` | mp3 in tmp | download succeeded | redownload on retry (we don't keep mp3s long-term) |
| `transcribe` | `<video_id>.json` + `.meta.json` | transcript saved | `podsave retry <video_id>` |
| `extract` | — (note is atomic write) | extraction saved to note | `podsave retry <video_id>` |
| `render` | note at vault path | — | trivial, local-only |

A failed run appends a `status: "failed"` or `status: "transcript_only"` row to `processed.jsonl`. `podsave retry <video_id>` finds the last transcript-only run for that ID and re-runs extraction.

The mp3 is transient: downloaded to `~/.podsave/tmp/`, deleted after transcription succeeds. No point keeping a 200MB file when we have the transcript.

## Versioning

When writing `YYYY-MM-DD - Channel - Title.md`, check for existing file:
- Not present → write v1, frontmatter `version: 1`.
- Present → scan for `Title (v2).md`, `Title (v3).md`, ... write the next integer.

Never overwrite. The `processed.jsonl` log records which version each run produced.

## CLI Surface

```
podsave <url>                    Process a single URL now
podsave --dry-run <url>          Print estimated cost, don't spend money
podsave --drain                  Process queue sequentially
podsave queue add <url>          Append URL to ~/.podsave/queue.txt
podsave queue list               Print the queue
podsave retry <video_id>         Re-run extraction from cached transcript
podsave init                     Create ~/.podsave/, prompt for API keys

Global flags:
  --force                        Bypass duration guards (< 15m or > 4h)
  --vault-path PATH              Override vault path for this run
  --model NAME                   Override OpenAI model
```

CLI design using `create-cli` conventions (short help text, `--dry-run` is discoverable, subcommand grouping for queue ops). Let the plan phases shake out the exact Typer structure.

## Testing Approach

- **Unit**: every `storage/*`, every `utils/*`, `render.note`, `filenames.next_version_path`. Fast, no network.
- **Pipeline**: mock `assemblyai` and `openai` clients at their boundaries; test pipeline stages pass correct data through.
- **Smoke**: CLI help + config-missing error + playlist rejection. Fast, no network.
- **Integration** (opt-in via `PODSAVE_INTEGRATION=1`): one real short YouTube video, asserts transcript + note land. Run manually before shipping each phase; not in pre-push.
- **Pre-push**: `uv run ruff check .` + `uv run pytest -q` (unit + smoke only).

Red/green TDD on anything data-shaped: models, filename sanitizer, versioning, queue/log, render output. Skip TDD for thin wrappers over SDKs (download.fetch, transcribe.run) — test them with mocks after.

## Open Questions / Decisions Punted to Implementation

1. **OpenAI model choice.** Default `gpt-4.1` or `gpt-5`? Validate quality on a fixture transcript in Phase 4. User can override with `--model`.
2. **Prompt tuning.** First draft goes in `extract_v1.md`. After dogfooding 3 videos, we'll know if it needs a rewrite → `extract_v2.md`. Version field in `ExtractionResult` means we can tell which prompt generated which note later.
3. **Speaker label resolution.** AssemblyAI returns `"A"`, `"B"`, `"C"` by default. For Phase 4, render as-is. Nice-to-have: prompt the extraction model to guess real names from context and pass them through to the render — punt to v1.1.
4. **Timestamp granularity in quotes.** AssemblyAI gives millisecond start times per utterance. Round to the second for the YouTube `t=` link. Show `MM:SS` in the displayed label (or `H:MM:SS` for long videos).
5. **What counts as a "spicy take" vs "insight"?** The prompt will define this. Rough heuristic: spicy take = contrarian / provocative claim; insight = useful non-obvious idea; quote = verbatim worth-remembering line. The model will sometimes disagree; that's fine, we can iterate.
6. **Cleanup of old mp3s on crash.** Add a `podsave doctor` command later to clean `~/.podsave/tmp/` orphans. Not v1.

## Risks

- **AssemblyAI pricing shifts or quality regression.** Mitigation: pricing is configurable in `utils/cost.py`; the extract provider is already abstracted as a Protocol, so swapping is cheap.
- **YouTube auth / rate-limiting on yt-dlp.** Mitigation: yt-dlp is well-maintained; if it breaks, the error surfaces cleanly and we wait or update the dep.
- **Long videos (> 1h) blow the OpenAI context window or produce stale extractions by the end.** Mitigation: transcript is well within modern context windows (3h ≈ 36k tokens, fits in every serious model). If quality degrades on long content, chunk+merge in v1.1.
- **Obsidian filename collisions / weird unicode in channel names.** Mitigation: aggressive sanitizer in `filenames.py` with a fixture-based test suite covering emoji, CJK, slashes, colons.
- **Prompt injection via video content.** Low stakes (it's a personal tool writing to a local vault), but a malicious transcript could try to manipulate the extraction. Not mitigating in v1; worth being aware.
