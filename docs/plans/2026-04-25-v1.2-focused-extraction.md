# podsave v1.2 — Focused Extraction (`--focus`)

A single feature: let the user re-run extraction through a lens. The same cached transcript can produce a "career advice" note, a "AI policy claims" note, and a broad note — each landing as its own file, sharing the expensive STT cache, costing only the extraction tokens (~$0.01–0.03 per pass).

## Goal

```bash
podsave save <url>                          # default, broad extraction (unchanged)
podsave save <url> --focus "career advice"  # focused on first pass
podsave retry <video_id> --focus "memory"   # second lens, no STT spend
podsave retry <video_id> --focus "ai policy"  # third lens, also free of STT spend
```

After this lands, the natural workflow is: broad save once, then re-extract through whatever lens you care about as you process the note.

## Why this is the next step

Three things make `--focus` the right v1.2 over the bigger bets (cross-note retrieval, weekly digest):

1. **It rides existing infrastructure.** The transcript cache, prompt-versioning, and `(v2)/(v3)` filename collision logic already make re-extraction cheap and addressable. Speaker-name resolution (v1.1) just hardened the same loop end-to-end.
2. **It's contained.** No new external services, no new storage shape, no embedding/index layer. One new prompt section, one new frontmatter field, one CLI flag, one filename rule.
3. **It compounds with the past.** Every cached transcript I already have (5 dogfood videos plus future ones) becomes immediately useful through new lenses without re-paying STT.

Cross-note retrieval and the weekly digest are bigger structural choices that benefit from waiting until the vault has more material — and `--focus` is the cheapest way to grow the vault per video.

## Non-goals (v1.2)

- Cross-vault retrieval / semantic search across notes (still v2 thesis).
- Multiple foci in one run (`--focus "career,sleep"`). If you want both, run twice.
- Saved focus presets (`podsave focus add "ai-policy" "claims about AI policy and..."`). YAGNI until I'm typing the same focus three times.
- Auto-suggesting focus areas from the transcript. Cute, premature.
- Diff/merge across focused notes. They live as separate files; Obsidian's MOC pattern handles aggregation.

## CLI surface

```bash
podsave save <url> [--focus TEXT] [--dry-run] [--force]
podsave retry <video_id> [--focus TEXT]
```

`drain` does **not** get `--focus` — bulk processing of a queue with one focus doesn't match how the queue is actually used (heterogeneous URLs added at different times for different reasons). If a real use case shows up, add it later.

`--dry-run` ignores `--focus` for cost preview purposes — the cost delta is rounding error.

## What "focus" means

A free-form string the user supplies. The extraction prompt appends a short addendum:

> Additionally, the user has asked you to focus this extraction on: **{focus}**.
>
> Apply your normal quality bar (verbatim quotes, non-obvious insights, no padding) — but only return items that are clearly relevant to this focus. If the video does not meaningfully address this focus, return zero items rather than padding with off-topic picks.
>
> Speaker resolution still applies regardless of focus.

Key prompt rules:

- The model can return **0 items** if the focus doesn't fit the content. Better than off-topic padding.
- The 10-item cap still applies.
- Quote / insight / spicy_take kinds still apply.
- Speakers map still returned (it describes the transcript, not the items).

## Output changes

### Frontmatter

Add one optional field:

```yaml
focus: "career advice"
```

Omit the line entirely when no focus is set (don't write `focus: ` blank — keeps default broad notes visually identical to today's output).

### Tags

Append a focus sub-tag nested under `podsave/` for Obsidian browsing. Slugify the focus the same way as for filenames:

```yaml
tags:
  - podsave
  - podsave/career-advice
```

Keeping it under the `podsave/` namespace means all focused notes show up grouped in Obsidian's tag pane (`#podsave/career-advice`, `#podsave/sleep`, etc.) rather than scattered alongside unrelated tags. Default broad notes keep just `[podsave]`.

### Filename

Append `(focus: <slug>)` before the extension when focus is set:

- Broad: `Channel — Title [2026-04-25].md`
- Focused: `Channel — Title [2026-04-25] (focus: career-advice).md`
- Re-running the same focus: `… (focus: career-advice) (v2).md`

Slug rules (in `src/utils/filenames.py::focus_slug`):

- Lowercase, NFC, replace `[^a-z0-9]+` runs with `-`, trim leading/trailing `-`, cap at 40 chars.
- Empty or whitespace-only focus → treat as no focus (no suffix, no tag, no frontmatter line).

Why include focus in the filename instead of relying on `(v2)`: scanning the vault directory should make it obvious what each file *is*. Versioning alone forces opening notes to find out.

## Data model changes

All additive, no migration:

- `ExtractionResult.focus: str | None = None` — what was passed in (raw, untruncated). Recorded for completeness even though render reads it from the runtime path.
- `RunRecord.focus: str | None = None` — so `processed.jsonl` shows the lens for each entry.
- No change to `Insight`. The model returns the same item shape; the prompt narrows what counts as a top item.

## Pipeline wiring

```
extract.extract(raw, meta, *, api_key, model, focus: str | None = None) → ExtractionResult
```

When `focus` is set:
- Append the focus addendum to the system prompt (don't replace it).
- Stuff `focus` into the returned `ExtractionResult.focus`.

`render.render_note(meta, extraction, *, version, processed_at, cost_usd)` reads `extraction.focus` directly — signature stays the same.

CLI helpers in `src/cli.py`:
- `_extract_render_and_log(meta, raw_transcript, *, stt_cost, cfg, console, focus=None)` — threads `focus` into the extract call and the `RunRecord`.
- `_process_url(meta, *, estimate, force, console, focus=None)` — passes through.
- `save` and `retry` accept `--focus` and pass to the helpers.

## Edge cases worth pinning down

- **Empty or whitespace focus**: treated as unset everywhere (no flag, no slug, no tag, no frontmatter, no filename suffix).
- **Focus with quote chars or YAML-breaking chars**: `_yaml_quote` already escapes `"` and `\`. The slugifier strips everything non-alphanumeric, so the filename is safe.
- **Same focus run twice**: standard `(v2)` versioning kicks in, scoped to the focused basename.
- **Broad save after focused save**: writes the broad basename — no collision. Same for focused after broad.
- **Two different foci on the same video**: each lands as a separate file with a distinct focus slug.
- **Long focus string** (e.g., a paragraph of instructions): filename slug truncated to 40 chars, frontmatter stores the full string.

## Risks

- **Prompt regression**: The focus addendum could leak into the broad-extraction case if the prompt isn't carefully gated. Mitigation: the addendum is appended in code only when `focus` is non-empty; tests cover both branches.
- **Quality on tight foci**: If the user supplies a hyper-narrow focus ("the word 'flywheel' specifically"), the model will likely return 0 items. Behavior: the CLI gracefully refuses to write a note in that case, raises a typed `EmptyExtractionError`, prints a short actionable message ("no items matched focus '<focus>' — try broader phrasing or run without --focus"), and exits 1. The audit trail still records a `RunRecord(status="failed", error=...)` so `processed.jsonl` shows the attempt and `podsave stats` counts it as a failed run. The run is auditable without polluting the vault with empty-content placeholder notes.
- **Slug collisions**: Two different foci that slugify to the same string ("AI policy" and "ai-policy") collide on filename. Standard `(v2)` versioning handles it; user can disambiguate via the `focus:` frontmatter field. Acceptable.
- **Prompt token growth**: Negligible. The addendum is ~6 lines. The model already reads the whole transcript.

## Cost story

Same model, same transcript, ~same output bound. Each focused retry costs the standard extract spend (pennies). No new cost dimension. `RunRecord.cost_usd["extract"]` already captures this; `podsave stats` will naturally show focused retries in the "retries" count.

Same red/green TDD pattern as v1.1 phases. Each phase ends behind a green test bar before the next.

1. **Phase A — slugification + filename** (pure functions, easiest wins).

6. **Docs** — `FEATURES.md` (new flag + filename rule + frontmatter field), this plan flipped to `complete`, `2026-04-23-podsave-v1.md` left alone (focus is additive to v1, doesn't resolve any of its rough edges).

Estimated total: comparable to v1.1 (a focused session). The TDD phases are smaller because there's no new command surface — just a flag threaded through existing code.

## Resolved design decisions

1. **Zero-item case**: gracefully refuse to write the note. Raise `EmptyExtractionError`, print an actionable message, exit 1, log a `status="failed"` RunRecord with the focus and reason. No empty placeholder notes in the vault. This guards both the focused and unfocused paths.
2. **`drain --focus`**: not added. The queue is heterogeneous (URLs added at different times for different reasons); a single global focus doesn't match the use case. Save / retry only.
3. **Focus does not affect speaker resolution**: the speakers map describes who's in the transcript, not which lens is being applied. The model still returns the full speakers list regardless of focus.
4. **Tag namespace**: `podsave/<slug>` (nested under the parent `podsave` tag), so all focused notes show up grouped in Obsidian's tag pane.

## Open questions to resolve during build

1. **Do we need `--prompt-version` to opt back to v1?** Probably no — focus rides on top of v2. If quality regresses for some focus, we'd ship a v3 prompt for everyone.

## What this unlocks

After v1.2 ships, the next decision point is bigger:

- **Cross-note retrieval (v2 thesis)** — the case strengthens the more focused notes the vault has.
- **Weekly digest** — could naturally pair with focused extraction ("digest of my AI-policy lens this week").

Either path benefits from `--focus` first. So shipping v1.2 isn't just utility — it's a cheap call option on whichever direction v2 takes.
