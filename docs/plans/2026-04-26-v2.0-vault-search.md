# podsave v2.0 — Vault search (`podsave search`)

A new command that turns the vault from a write-only archive into a searchable corpus. Substring/regex search across every callout in every podsave-tagged note, sliced by frontmatter (kind, channel, focus, recency), output to terminal or to a fresh "callouts MOC" note inside the vault.

## Goal

```bash
podsave search "memory consolidation"
podsave search "flywheel" --kind quote
podsave search "agency" --channel Anthropic --since 90d
podsave search "career" --focus "career advice" --write
```

Each invocation walks the vault, finds callouts whose text matches the query, applies filters, ranks results, and either prints them to the terminal (default) or re-renders them into a new search-results note in the vault.

## Why this is the next step

Three v1 → v1.2 design choices set this up cleanly, so it's the cheapest natural next step:

1. **Curation already happened.** The whole pipeline exists *because* raw transcripts aren't browseable; callouts are the readable unit. Search over callouts means every hit is already a self-contained, attributed, time-stamped moment — not a 30-second dive into context.
2. **Frontmatter is typed.** Every note already carries `channel`, `published`, `focus`, `tags: [podsave, ...]`. Filters land as ~5-line frontmatter parses, not a new index.
3. **Each focused note is its own row.** v1.2 made it normal to have 2–3 versions of the same video under different lenses. Search is what makes those re-extractions actually compound — without it, focused notes are just more clutter.

This is also the cheapest test of "is the vault corpus useful in aggregate?" before investing in embeddings, weekly digests, or cross-vault search.

## Non-goals (v2.0)

- **Embedding/semantic search.** Designed-for, not shipped here. The Matcher protocol leaves a clean seam.
- **Searching cached transcripts.** Defeats the curation; raw STT is intentionally not browseable.
- **Searching outside the podsave corpus.** Only notes that carry `podsave` in `tags:` are eligible. The vault may have other content; we don't touch it.
- **Stored search indexes.** No `.podsave/index.db`, no incremental update. The vault is small (10s–100s of notes); a fresh walk per query is fine for now.
- **Saved searches / aliases.** YAGNI. Use shell history.
- **Updating an existing `--write` note.** A second invocation writes a new dated file; we don't try to merge or version the results note. (Unlike video notes, search results are ephemeral views; let the user delete or replace.)
- **Cross-callout summaries.** "Summarize the top 10 hits" is a digest concern, not a search concern. Search returns the matching callouts verbatim.

## CLI surface

```bash
podsave search QUERY
              [--kind quote|insight|spicy_take]
              [--channel TEXT]
              [--focus TEXT]
              [--since DATE]            # YYYY-MM-DD or relative: 7d / 30d / 1y
              [--limit N]               # default 20
              [--write]                 # also write a results note into the vault
```

`QUERY` is passed through to the matcher. The default `GrepMatcher` does case-insensitive substring matching for each whitespace-separated token (matching all tokens AND-style). Phrase queries work by quoting at the shell level. No regex flag in v1; if rg is the matcher, regex characters in the query are escaped before being passed.

`--limit` defaults to 20. `0` means unlimited (useful with `--write`).

## What "search" matches

For each candidate callout, we test the query against the **callout text body** (the `text` field) and **the optional context line** (`context` field). Speaker letter and rank are not searched — they're metadata, not content.

A callout matches a query when **every whitespace-separated token in the query** appears in the searched text (case-insensitive). Multi-token queries are AND. Tokens are matched as substrings (so `bench` hits `benchmark`).

Why AND-of-tokens, not exact phrase: it's the matching mode that survives paraphrasing best for short queries, which is what the user is realistically typing. Phrase matching is still available — quote the whole query and your shell preserves it as one token.

## Output format

### Terminal (default)

Print a Rich `Group` of result panels, one per matching callout, ranked. Each panel:

```
┌─ Anthropic — AI prompt engineering [2024-09-05] ──────────────────┐
│ > [!quote] 2. Quote — David Hershey @ 23:14                      │
│ > "If you can't explain the task to a competent outsider, your   │
│ >  prompt isn't ready."                                          │
│ ↪ https://www.youtube.com/watch?v=T9aRN5JkmL8&t=1394s            │
└──────────────────────────────────────────────────────────────────┘
```

- Header line: source note basename (no extension, no `.md`) so it's grep-friendly.
- Body: callout body, untruncated (if it's huge, that's a feature — search results give you the full quote, not a snippet).
- For quotes: explicit timestamp URL on its own line (so you can `cmd-click` in iTerm/Warp without rg-style line-truncation noise).

Footer line under all panels: `N callout(s) across M note(s) — query: "<query>"` plus filter summary if any filters were active.

### Vault note (`--write`)

Write `<vault>/Callouts/Search — <query-slug> [YYYY-MM-DD].md`. Slug rules: `focus_slug(query)` (lowercase, non-alnum → `-`, 40-char cap). The note body re-renders the matching callouts using the same `_callout` formatter as the main pipeline (so they look exactly like first-class callouts in Obsidian) and adds a `[[Source Note Title]]` wikilink under each one for navigation.

Frontmatter:

```yaml
---
title: 'Search — "memory consolidation"'
query: "memory consolidation"
generated: 2026-04-26T18:30:00
matches: 7
notes_searched: 23
filters:
  kind: quote
  channel: null
  focus: null
  since: null
tags:
  - podsave
  - podsave/search
---
```

The `podsave/search` sub-tag groups all written results in Obsidian's tag pane, parallel to `podsave/<focus-slug>` for focused video notes.

The `Callouts/` subdirectory is created lazily on first `--write`. It lives under the existing vault path (no new config).

## Data flow

```
vault scan → NoteIndex (parsed notes + callouts)
           ↓
        filters (kind/channel/focus/since)  → narrowed candidate list
           ↓
       matcher.find(query, candidates)      → list[Match(callout, score)]
           ↓
        rank + truncate to --limit          → ranked Match list
           ↓
   render_terminal()  or  render_vault_note()
```

`NoteIndex` is built per-invocation, no persistence. For ~100 notes with ~10 callouts each, the full pipeline should run in well under a second.

## Module layout

A new `src/search/` subpackage. The pipeline analogue (`src/pipeline/`) is for the YouTube → note flow; search is a separate read-side capability.

```
src/search/
  __init__.py
  index.py     # NoteIndex: parse a vault dir → list[Note] (frontmatter + callouts)
  callout.py   # Callout dataclass: source_note, kind, rank, title, text, context, speaker, start_ms, url
  matcher.py   # Matcher protocol; GrepMatcher (rg if installed, else Python re fallback)
  filters.py   # NoteFilter dataclass + apply_filters()
  ranker.py    # rank(matches, query) → ordered list (term-count primary, recency tiebreak)
  render.py    # render_terminal(); render_vault_note()
```

Plus:
- `src/cli.py` — new `search` command wiring the above
- `tests/test_search_*.py` — one file per module

The `Matcher` protocol is the seam for embedding mode later:

```python
class Matcher(Protocol):
    def find(self, query: str, callouts: list[Callout]) -> list[Match]: ...

class Match(BaseModel):
    callout_id: str          # source note path + callout rank, deterministic
    score: float             # higher = better
    matched_terms: list[str] # for highlighting + debug
```

A future `EmbeddingMatcher` implements the same protocol; the rest of the pipeline doesn't change.

## Note parsing strategy

Render is the spec; parsing has to invert it. Two-pass:

1. **YAML frontmatter** between the first two `---` lines. Use the `tomli`-style approach but YAML — pull in `pyyaml` or do a tiny hand-parse since the schema is fully known and stable. Lean: `pyyaml` (small, well-trodden, already on most systems; one dependency to add the rest of v2 will likely also want).
2. **Callout extraction**: a callout starts with `> [!<type>] <title>` and continues for every following line that starts with `>`. Parse each block back into `Callout(kind, rank, title, body_text, context)` using the inverse of the render rules:
   - First body line that starts and ends with `"` → quote text (strip the quotes).
   - Otherwise → text.
   - A trailing `*italic*` line → context.
   - `kind` from `[!type]` (`note` → `insight`, `quote` → `quote`, `warning` → `spicy_take`).
   - `rank`, `speaker`, `start_ms`, `url` parsed from the title line via regex.

Render and parse share format constants from `src/pipeline/render.py` to keep them locked together.

## Edge cases worth pinning down

- **Empty vault**: `0 callouts across 0 notes`, exit 0.
- **Note with zero callouts**: shouldn't exist after v1.2 (CLI refuses to write empty-body notes), but parse defensively — returns a `Note` with `callouts: []`. Filters/matchers don't care.
- **Note authored by hand / hand-edited**: if the user adds a callout manually, parse will still pick it up provided it follows the format (`> [!type] N. Label — ...`). Best-effort.
- **Long quote text**: rendered untruncated. Might overflow narrow terminals; that's acceptable — Rich wraps cleanly.
- **Query that's only filters** (e.g. `podsave search "" --channel Anthropic`): treat empty query as "match everything"; effectively a vault filter. Useful for `--write` to compile every Anthropic callout into one note. Decision: allow it.
- **`--write` collision**: same query/date already on disk → standard `(v2)` versioning kicks in via the existing `next_version_path` helper.
- **Subdirectories in vault**: walk recursively under `vault_path`; respect any `.obsidian/` or `.git/` directories by skipping them.
- **rg not installed**: matcher falls back to Python `re`; print a one-line dim notice on first run if `--debug` (hold on this, see Open Questions).

## Risks

- **Parser drift from renderer**: if render's format changes and parse doesn't, search silently misses callouts. Mitigation: round-trip test in Phase A is load-bearing; both modules live on shared format constants.
- **Hand-edited notes**: same risk in the other direction — a hand-edited note that no longer round-trips will silently drop from search. Mitigation: parse is permissive (best-effort); if it fails on a note, log a warning to stderr but keep going (don't crash the whole search).
- **AND-of-tokens isn't what some users expect**: people sometimes want OR. Mitigation: document clearly; if it bites, add `--any` later.
- **Case insensitivity is implicit**: makes "Apple" and "apple" the same. For 99% of queries this is right; for code-like queries it isn't. Punt to v2.1 if it shows up.
- **Vault contains non-podsave content**: filtered by the `podsave` tag check, so search doesn't accidentally crawl a Daily Notes folder. Documented in non-goals.
- **`pyyaml` adds a dependency**: lightweight, ubiquitous, but it's a new line in `pyproject.toml`. Acceptable.

Estimated total: similar shape to v1.2 — a focused session per phase. Phase A is the largest because parse has to invert render carefully. Phase B has the most surface area (two matcher impls). C/D/E are small.

## Resolved design decisions

1. **Result unit: callout, not note.** Callouts are the curated unit; rendering them as the result is what makes search qualitatively different from `rg` over the vault.
2. **Output: terminal default + `--write` for vault note.** Terminal for browsing, `--write` for keep/share/MOC. Same content, two presentations.
3. **Scope: notes only.** Cached transcripts are intentionally not searched. Easy to add later; impossible to take back without disruption.
4. **Matcher: pure-Python `GrepMatcher` behind a `Matcher` protocol.** Originally specced rg-with-Python-fallback, but since callouts are already in memory after `index.walk_vault`, shelling to rg adds complexity (temp file, subprocess, output parsing) without any speed benefit at this scale. The "grep" intent (substring, not embeddings) is preserved; the Matcher protocol still gives the embedding-later seam.
5. **Filters in v1: `--kind`, `--channel`, `--focus`, `--since`.** Each is a small frontmatter check; they cost ~nothing extra together. Slicing is the main reason cross-vault search is interesting.
6. **No persistence.** Walk the vault per invocation. Vaults are small; rebuilds are <1s; complexity of a stored index is not earned yet.
7. **`--write` to `<vault>/Callouts/`.** New subdirectory under the existing vault path. Sub-tag `podsave/search`. Standard `(v2)` versioning for collisions; no merge-into-existing logic.
8. **Empty query is allowed** when filters are present. `podsave search "" --channel Anthropic --write` becomes a valid "compile everything from this channel" workflow.

## Open questions to resolve during build

1. **Should we surface a dim `(rg not found, falling back to slower Python matcher)` notice when the fallback fires?** Probably yes, once per run, to stderr. Confirm during Phase B.
2. **Does `--write` need a `--no-confirm` flag for collisions?** Default versioning behavior (`(v2)`) handles it implicitly; might not need anything explicit. Confirm during Phase D.
3. **Should the vault-note frontmatter `query:` field always quote the raw query, or normalize whitespace?** Lean: store raw, render-time normalization is fine.

## What this unlocks

After v2.0 ships, two natural follow-ons sit one level up:

- **Embedding mode (`v2.1` candidate)** — drop a `SentenceTransformerMatcher` next to `GrepMatcher`, behind the same protocol. Pre-compute embeddings into a sidecar file per note (`<vault>/.podsave/embeddings/<basename>.npy` — gitignored). Toggle via `--mode embed`, default still grep. Decision point: only worth it if grep mode is *frequently* missing the conceptual variant the user wanted.
- **Digest mode** — `podsave digest --since 7d --focus X --write` reduces a search result set into a markdown brief. Composes naturally with this command's filters; the synthesis step is a thin OpenAI call over the matching callouts. Especially interesting once `~/Dev/feed`-style weekly cadence is on the table — there's a clear shape where podsave produces the callouts and a separate piece composes the brief.

Both are smaller than v2.0 once this lands; v2.0 is doing the load-bearing infrastructure work.
